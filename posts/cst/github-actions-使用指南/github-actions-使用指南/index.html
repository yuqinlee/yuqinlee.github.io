前言#

本博客基于 Astro 框架的 Fuwari 模板构建，源码托管在 yuqinlee/yuchin-space，通过 GitHub Pages 对外提供服务。

每次写完文章，只需要一个 git push，剩下的构建和部署全部由 GitHub Actions 自动完成。这篇文章就以此博客为例，带你从零搭建一套完整的「Push → 自动部署到 GitHub Pages」工作流。

基础概念#

什么是 GitHub Actions？#

GitHub Actions 是 GitHub 内置的 CI/CD（持续集成 / 持续部署） 服务。你可以把它理解为一个免费的自动化机器人——当仓库发生特定事件（如 push、PR、定时触发）时，它自动执行你预先定义的任务。

  • 免费额度：公开仓库无限使用，私有仓库每月 2000 分钟
  • 配置方式：在 .github/workflows/ 目录下编写 YAML 文件
  • 运行环境：GitHub 提供 Windows / macOS / Ubuntu 虚拟机

什么是 GitHub Pages？#

GitHub Pages 是 GitHub 提供的 免费静态网站托管服务。它直接托管仓库中的 HTML/CSS/JS 文件，适合博客、文档站、项目主页等静态内容。

  • 免费，无限流量
  • 支持自定义域名
  • 自动 HTTPS
  • 两种发布方式：
    1. 从特定分支发布（如 gh-pages 分支）
    2. 从 GitHub Actions 部署（推荐，更灵活）

整体架构#

核心思路：源码和构建产物分开存放。

仓库分支内容
yuqinlee/yuchin-space（源仓库）yuchin-space博客源码（Markdown、组件、配置等）
yuqinlee/yuqinlee.github.io（目标仓库）master构建产物（纯 HTML/CSS/JS）

准备工作#

1. 创建仓库#

你需要两个 GitHub 仓库：

  • 源仓库：存放博客源码，名称随意（如 my-blog）
  • 目标仓库：用于 GitHub Pages，名称必须是 <你的用户名>.github.io，例如 yuqinlee.github.io

注意：目标仓库必须是 Public（公开仓库），否则 GitHub Pages 需要付费计划。

2. 生成 SSH 部署密钥#

GitHub Actions 需要向目标仓库推送代码，这需要认证。最安全的方式是使用 SSH Deploy Key。

3. 配置 Deploy Key 和 Secrets#

目标仓库（<user>.github.io）：

  1. 进入 Settings → Deploy keys
  2. 点击 Add deploy key
  3. Title：GitHub Actions Deploy
  4. Key：粘贴 ~/.ssh/github-pages-deploy.pub 的全部内容
  5. ⚠️ 勾选「Allow write access」（否则无法推送）
  6. 点击 Add key

源仓库（博客源码）：

  1. 进入 Settings → Secrets and variables → Actions
  2. 点击 New repository secret
  3. Name：DEPLOY_KEY（名字随意，后续 workflow 中引用）
  4. Secret：粘贴 ~/.ssh/github-pages-deploy 私钥的全部内容（包括开头的 -----BEGIN OPENSSH PRIVATE KEY----- 和结尾）
  5. 点击 Add secret

各步骤详解#

  1. on.push.branches — 指定触发分支。只有推送到 yuchin-space 分支才会触发部署。可以根据需要改为 main 或其他分支名。

  2. concurrency — 并发控制。如果不设置，快速连续 push 两次会产生两个并行任务，后完成的可能覆盖先完成的。cancel-in-progress: true 确保总是部署最新代码。

  3. runs-on: ubuntu-latest — GitHub Actions 的运行环境。免费，性能足够构建静态站点。

  4. actions/checkout@v4 — 官方 action，将源码克隆到运行环境中。

  5. actions/setup-node@v4 — 安装指定版本的 Node.js。Astro 5 要求 Node.js >= 20。

  6. pnpm/action-setup@v4 — 安装 pnpm 包管理器。run_install: false 表示不自动安装依赖（我们手动控制）。

  7. pnpm install —frozen-lockfile — 严格按照 pnpm-lock.yaml 安装依赖，确保 CI 环境与本地一致。

  8. pnpm build — 执行构建。本博客的构建命令是 astro build && pagefind --site dist，输出到 dist/ 目录。

  9. peaceiris/actions-gh-pages@v4 — 第三方 action，将指定目录推送到目标仓库。

    • deploy_key：SSH 私钥，用于认证
    • external_repository：目标仓库（不同于当前仓库时需要指定）
    • publish_dir：要推送的目录
    • publish_branch：目标分支

常见问题与排查#

Q1：部署失败，提示 “Permission denied (publickey)”#

原因：SSH 密钥配置不正确。

排查：

  1. 检查 Secrets 名称是否与 workflow 文件中一致（区分大小写）
  2. 确认私钥内容完整（包括头尾的 -----BEGIN... 和 -----END...）
  3. 确认公钥已添加到目标仓库的 Deploy Keys 并勾选了 Allow write access
  4. 确认使用的密钥类型是 ed25519（某些旧环境可能不支持）

Q2：部署成功但页面不更新#

原因：GitHub Pages 有缓存，或者分支配置不正确。

排查：

  1. 进入目标仓库 Settings → Pages，确认 Branch 设置与 workflow 中 publish_branch 一致
  2. 等待 1-2 分钟让 CDN 缓存刷新
  3. 用隐私模式（无痕模式）打开页面，排除浏览器缓存
  4. 检查目标仓库的 master（或 main）分支是否确实收到了新提交

Q3：构建失败，提示 pnpm 版本不匹配#

原因：项目 package.json 中指定了 packageManager 字段，本地和 CI 环境 pnpm 版本不一致。

Q4：astro build 成功但 pagefind 报错#

原因：Pagefind 需要 dist/ 目录中的 HTML 文件作为输入，如果构建不完整可能出错。

解决：确保 pnpm build 命令正确：astro build && pagefind --site dist。如果 pagefind 找不到 dist/ 目录，检查 astro build 的输出路径配置（在 astro.config.mjs 中）。
